审查意见表模板

模板说明
本模板基于客户历史报告格式制定，用于指导报告整理员将《事实核查报告》和《债权分析报告》整合为标准化审查意见表。

⚠️ 格式要求（MANDATORY - 零容忍）

**本模板是纯文本段落格式，NOT Markdown格式！生成最终报告时必须进行格式转换！**

✅ 允许的格式：
- 完整的自然语言段落
- 章节标题（一、二、三、四、五、六、七）
- 自然段落内的叙述性文字

❌ 严格禁止的格式（客户交付标准）：
- Markdown标题语法（##、###、####等）
- 项目符号（-、*、•、○等）
- Markdown加粗语法（**文字**）
- Markdown表格语法（| --- |）
- 任何Markdown特殊符号

⚠️ 重要提示：模板本身就是最终格式

**本模板（第67-121行）已经是纯文本格式，这就是最终报告应该的样子！**

模板标题格式（第67-68行）：
```
[债权人名称]        ← 第1行：纯文本（不是 # [债权人名称]）
债权审查意见        ← 第2行：纯文本（不是 # 债权审查意见）
```

**错误理解**：认为模板需要"转换为纯文本"
**正确理解**：模板本身就是纯文本，直接使用这个格式

**Agent常见错误**：
```markdown
# 江苏兴洋船舶设备制造有限公司债权审查报告   ← ❌ Agent自己添加了 #
```

**正确做法**：
```
江苏兴洋船舶设备制造有限公司债权审查报告    ← ✅ 与模板格式完全一致
```

**生成指令**：
1. 读取模板第67-121行
2. 保持格式完全不变
3. 替换 `[占位符]` 为实际内容
4. 保存时确认无任何 Markdown 语法

格式转换示例：

错误示例（Markdown格式 - 禁止）：
```
## 二、合同签订情况

**主合同**：2024年1月1日，债务人与债权人签订...

- 合同编号：XYZ-2024-001
- 合同金额：1,000,000.00元
- 付款期限：2024年12月31日
```

正确示例（纯文本段落格式 - 要求）：
```
二、合同签订情况

2024年1月1日，债务人与债权人签订借款合同，合同编号为XYZ-2024-001，约定借款金额1,000,000.00元，付款期限为2024年12月31日。
```

格式验证规则：
```bash
# 最终报告中不得出现以下符号：
grep -E "^(##|###|-|\*\*)" final_report.md  # 结果应为空
```

## ⚠️ 模板使用强制规则 (MANDATORY)

### 占位符替换协议

**本模板存在两类占位符，处理方式不同：**

**第1类: 基本信息占位符** (第96-150行)
- `[债权人名称]`、`[年月日]`、`[金额]` 等
- **数据源**: 事实核查报告 + 债权分析报告
- **替换规则**: 见下文"替换决策规则"

**第2类: 内容占位符** (第105-149行)
- `[合同完整名称]`、`[违约责任条款的具体描述]`、`[具体审查情况描述]` 等
- **数据源**: 直接从技术报告提取原文
- **核心原则**: 提取不改善，保留不美化

---

### 替换决策规则

**规则1: 信息追溯 (Traceability)**
- ✅ 必须能追溯到事实核查报告或债权分析报告的具体段落
- ✅ 使用技术报告的原文表述（完整段落或句子）
- ❌ 不可使用自己的概括、改写或润色

**规则2: 来源验证 (Source Verification)**
- ✅ 内容来源: 技术报告的正文叙述（事实核查、债权分析）
- ❌ 禁止来源:
  - 证据目录/清单
  - Agent自己的推测
  - "常识性"补充
  - 为了"流畅"而添加的过渡句

**规则3: 替换决策 (Replacement Decision)**

**情形A: 技术报告有明确内容**
```
✅ 完整照抄技术报告原文
   - 保持句式结构不变
   - 保持表述风格不变
   - 包括模糊、保留性表述（如"证据未明确记载"）
```

**情形B: 技术报告标注了缺失**
```
✅ 保留技术报告的缺失标注
   示例: 如事实报告写"[债权人未填写]"
   → 审查意见表也应标注"[债权人未填写]"或"未提供"
   ❌ 不可删除、不可用默认值填充
```

**情形C: 技术报告未涵盖该章节内容**
```
✅ 使用标准缺失标记:
   - [技术报告未涵盖此内容]
   - [需补充材料后完善]
   ❌ 不可自行编写"合理"内容
   ❌ 不可删除模板章节
```

### ❌ 严禁的占位符填充行为

**禁止1: 美化提取 (Beautification)**
```
❌ 技术报告: "债权人申报利息50万元，但未提供计算依据。"
   错误做法: "债权人申报利息500,000.00元。"
   ✅ 正确: 保留"但未提供计算依据"

理由: 删除不利信息改变了事实完整性
```

**禁止2: 逻辑串联 (Logical Bridging)**
```
❌ 技术报告片段A: "2024年1月放款100万"
   技术报告片段B: "2024年12月违约"
   错误做法: "2024年1月放款后，双方履约顺利，直至12月发生违约。"
   ✅ 正确: "2024年1月放款100万元。2024年12月发生违约。"

理由: "履约顺利"是Agent添加的推测
```

**禁止3: 默认值填充 (Default Value)**
```
❌ 占位符: [是否回函及回函内容]
   技术报告: 未提及是否回函
   错误做法: "债务人未回函"
   ✅ 正确: "[技术报告未记载是否回函]"

理由: "未回函"是推测（可能是报告未查，不代表事实未发生）
```

**禁止4: 简化复杂 (Simplification)**
```
❌ 技术报告: "利息计算存在争议，债权人申报按年利率6%计算，但合同约定为LPR+50BP，经计算应为4.85%。鉴于就低原则，确认债权人申报的金额。"
   错误做法: "利息按债权人申报金额确认。"
   ✅ 正确: 完整保留技术报告原文

理由: 简化隐藏了重要的审查逻辑
```

**禁止5: 规范用语 (Terminology Standardization)**
```
❌ 技术报告: "欠条"
   错误做法: 统一改为"借款凭证"
   ✅ 正确: 保留"欠条"

理由: 证据类型不同，法律效力不同，不可擅自替换
```

**禁止6: 补全缺失 (Gap Filling)**
```
❌ 占位符: [如有配偶知情同意等情况，一并描述。]
   技术报告: 未提及配偶
   错误做法: 删除此句（似乎不适用）
   ✅ 正确: 保留"[技术报告未涉及配偶情况]"或删除整段

理由: 删除可能掩盖应审查而未审查的事项
```

**禁止7: 格式统一 (Format Normalization)**
```
❌ 技术报告: 日期格式不统一（"2024-1-1" / "2024年01月01日"）
   错误做法: 统一为"2024年1月1日"
   ✅ 正确: 保持技术报告原格式（除非明显笔误）

理由: 过度干预可能引入错误（如1月误改为01月）
```

```bash

grep -o '\[.*\]' final_report.md | head -20

# 结果应仅包含:
✅ [债权人名称] - 正常，审查意见表的固定格式
✅ [技术报告未记载XX] - 正常，标准缺失标记
❌ [年月日] - 遗漏，未替换的占位符
❌ [金额] - 遗漏，未替换的占位符
```

**步骤2: 格式转换验证**
```bash
# 确保已转换为纯文本格式
grep -E "^(##|###)" final_report.md
# 预期: 无结果（否则存在Markdown标题）

grep -E "^(-|\*) " final_report.md
# 预期: 无结果（否则存在列表符号）

grep "\*\*" final_report.md
# 预期: 无结果（否则存在加粗语法）
```

**步骤3: 来源追溯验证**
```
随机抽取5个关键段落（如"合同签订情况""管理人审查结论"）:
□ 每段都能在技术报告中找到对应原文?
□ 表述是否与技术报告一致（未美化/简化）?
□ 是否包含技术报告的缺失标注（未删除）?
```

**步骤4: 禁止改善行为自查**
```
□ 是否存在技术报告未出现的"连接词"（如"经过"/"随后"/"因此"）?
□ 是否存在技术报告未出现的"评价性"词汇（如"合理"/"妥当"）?
□ 是否存在技术报告的模糊表述被"澄清"?
□ 是否存在技术报告的不一致被"协调"?

以上任一项为"是" → FAIL，需重新提取
```

### 📋 使用示例

**示例1: 正确的占位符替换**
```
占位符: [违约责任条款的具体描述]
技术报告原文: "合同第8条约定，如借款人未按期还款，应按逾期本金的日万分之五支付违约金。"

✅ 正确替换:
"合同第8条约定，如借款人未按期还款，应按逾期本金的日万分之五支付违约金。"

❌ 错误替换1（美化）:
"违约金按日万分之五计算。"
→ 删除了条款编号，丢失证据定位

❌ 错误替换2（补全）:
"合同第8条明确约定了违约责任：如借款人未按期还款，应按逾期本金的日万分之五支付违约金，该约定符合法律规定。"
→ 添加了"明确"和"符合法律规定"，属于Agent自己的评价
```

**示例2: 技术报告缺失的处理**
```
占位符: [如有补充协议，继续描述...]
技术报告: 未提及补充协议

❌ 错误做法（删除）:
删除这段占位符，似乎不适用

❌ 错误做法（推测）:
"双方未签订补充协议。"

✅ 正确做法（标注）:
"[技术报告未涉及补充协议情况]"

✅ 正确做法（删除整段前提）:
如果确定整个章节不适用（如无担保的债权），可删除整段"四、担保情况"
```

**示例3: 技术报告模糊表述的保留**
```
技术报告原文: "债权人提供了疑似放款凭证，但转账摘要为'往来款'，未明确标注为本案借款。"

❌ 错误做法（澄清）:
审查意见表写: "债权人提供了放款凭证，金额为XX元。"
→ 删除了"疑似"和证据瑕疵说明

✅ 正确做法（保留）:
审查意见表写: "债权人提供了疑似放款凭证，但转账摘要为'往来款'，未明确标注为本案借款。"
→ 完整保留原文的保留性表述
```

**示例4: 技术报告不一致的处理**
```
事实核查报告: "放款日期为2024年1月1日"
债权分析报告: "放款日期为2024年1月10日"

❌ 错误做法（选择看起来更可靠的）:
审查意见表写: "2024年1月10日放款"
→ Agent擅自选择，可能选错

❌ 错误做法（协调）:
审查意见表写: "2024年1月放款"
→ 用模糊表述掩盖矛盾

✅ 正确做法（保留矛盾）:
审查意见表写: "[技术报告存在不一致：事实报告记载1月1日，分析报告记载1月10日，需进一步核实]"
→ 暴露问题，促使修正
```

### ⚠️ 违规后果

**如检测到以下情况，报告将被退回重做：**
1. 无法追溯到技术报告原文的段落（来源不明）
2. 美化或简化了技术报告的表述（改善原文）
3. 删除了技术报告的缺失标注或保留性表述（掩盖问题）
4. 协调了技术报告之间的不一致（擅自选择）
5. 添加了技术报告未出现的连接词、评价词（自行发挥）
6. 使用了未替换的占位符（`[年月日]`、`[金额]`等）
7. 使用了Markdown格式（`##`、`-`、`**`等）

**核心原则**: 报告整理员是"搬运工"，不是"编辑"。忠实提取技术报告原文，不改善、不美化、不补全、不协调。

内容映射规则

从源报告到审查意见表的标准七章节映射：

一、债权申报情况：来源于事实核查报告申报情况表，提取申报日期、债权人信息、金额构成等要点。
二、合同签订情况：来源于事实核查报告债权发生情况查明，重点描述主合同条款内容，担保合同内容移至担保情况章节。
三、合同履行情况：来源于事实核查报告债权发生时间线，提取放款记录、还款记录、违约事实等关键信息。
四、担保情况：整合两份报告的担保信息，包括担保合同签订、质押登记、保证责任、担保现状等详细内容。
五、涉诉情况：来源于两份报告的诉讼信息，提取案件信息、判决内容、执行情况等要点。
六、债务人核查情况：使用标准化表述，说明债务人对申报债权的态度。
七、管理人审查结论：来源于债权分析报告审查确认部分，明确确认金额、债权性质、审减说明等结论。

客户特定要求

写作风格要求：每个章节使用完整的段落叙述，避免使用项目符号或列表（合同履行情况的编号除外），保持法律文书的正式、客观语气，确保内容流畅、逻辑清晰。

格式规范要求：金额使用阿拉伯数字并保留两位小数，日期采用YYYY年MM月DD日格式，法律文书完整引用案号，公司名称首次出现使用全称后续可用简称。

三、合同履行情况

1. [年月日]，[债权人简称]依约向[债务人简称]发放借款[金额]元，款项从[债权人账户信息]汇入[债务人账户信息]，履行了放款义务。

3. [年月日]，[债权人简称]向[债务人简称]发出《催收函》，要求其履行还款义务。[债务人简称][是否回函及回函内容]。
