# Report Organization Skill

## Overview

Consolidate technical reports from fact-checker and debt analyst into final standardized review opinion forms (审查意见表) for client delivery. This skill handles report integration, template application, file organization, and quality control.

## When to Use This Skill

- Consolidating fact-checking + debt analysis reports into 审查意见表
- Applying client-specific report templates
- Organizing and standardizing file naming
- Creating final deliverable packages
- Generating file inventories and document structures

## Prerequisites

- **Completed fact-checking report** from debt-fact-checker
- **Completed debt analysis report** from debt-claim-analyzer
- **Calculation files** generated by debt analyst
- **Client template** from `templates/review_opinion_form_template.md`
- **Verified dates** from `.processing_config.json`

## ⚠️ MANDATORY Pre-Work Check: Multi-Source Date Verification

**BEFORE starting report consolidation, you MUST:**

1. **Read Configuration**: Extract bankruptcy filing date from `.processing_config.json`
2. **Multi-Source Cross-Verify**:
   - Compare with fact-checking report dates
   - Compare with debt analysis report dates
   - Ensure ALL three sources match exactly
3. **Record Confirmation**: Document verified dates at report beginning
4. **Handle Inconsistencies**: If ANY mismatch found, STOP work immediately and report

**Example Output Format**:
```
✅ 破产受理日期核对完成
- 破产受理日期：2023-05-12
- 停止计息日期：2023-05-11
- 配置文件日期：2023-05-12 ✓
- 事实核查报告日期：2023-05-12 ✓
- 债权分析报告日期：2023-05-12 ✓
- 多重验证结果：一致
```

**Critical Importance**: As the final processing stage, report organizer has responsibility to ensure ALL dates in client-delivered reports are accurate. Any date error directly impacts client decision-making.

## Core Workflow (5-Step Process)

### Step 1: Receive and Verify Input Reports

**Objective**: Confirm all input materials are complete and consistent

**Actions**:
1. **Verify report receipt**:
   - Check fact-checking report exists in `工作底稿/`
   - Check debt analysis report exists in `工作底稿/`
   - Check calculation files exist in `计算文件/`

2. **Verify date consistency**:
   - Execute mandatory date verification (see above)
   - Confirm all three sources have matching dates

3. **Verify content completeness**:
   - Fact-checking report has complete timeline
   - Debt analysis report has final confirmation amounts
   - Calculation files generated for all calculated items

**Validation Checklist**:
```
□ Fact-checking report complete and readable
□ Debt analysis report complete with final amounts
□ Calculation files present (or explanation TXT if none)
□ Bankruptcy dates verified and consistent across all sources
□ No obvious errors or incomplete sections
```

### Step 2: Read and Understand Client Template

**Objective**: Understand template structure and requirements

**Template Location**: `templates/review_opinion_form_template.md`

**Actions**:
1. **Load template**:
   - Read current client's template file
   - Identify template version and client specifications

2. **Understand structure**:
   - Identify required sections (必填章节)
   - Identify optional sections (可选章节)
   - Note format requirements
   - Understand content mapping rules

3. **Extract mapping instructions**:
   - Read content mapping table from template
   - Understand which source sections map to which template sections
   - Note any client-specific terminology or style requirements

**Key Template Elements**:
- Section structure and hierarchy
- Content mapping rules (from technical reports to review form)
- Format specifications (tables, headings, numbering)
- Required vs. optional content
- Writing style guidelines

### Step 3: Extract and Map Content

**Objective**: Extract content from technical reports according to template mapping

**Core Principles**:

#### Principle 1: Single Source of Truth
- **ALL substantive content** must come from fact-checking and debt analysis reports
- **NO new analysis or conclusions** added by report organizer
- **Accuracy**: Do not modify original data or conclusions

#### Principle 2: Content Mapping
- **Follow template mapping table**: Each template section has defined source
- **Flexible matching**: Adapt to actual content structure
- **Traceability**: Maintain ability to trace back to source

#### Principle 3: Information Deduplication
- **Identify overlaps**: Recognize duplicate information in both reports
- **Keep most detailed version**: Retain final/complete version
- **Avoid repetition**: Same information should appear only once

**Extraction Strategy**:

**From Fact-Checking Report**:
- Creditor basic information (name, contact, registration)
- Declaration summary (declared amounts by category)
- Debt relationship timeline
- Evidence classification and hierarchy
- Key factual findings

**From Debt Analysis Report**:
- Independent debt relationship confirmation
- Amount item breakdown
- Interest calculation results
- Statute of limitations analysis
- Final confirmation amounts
- Debt classification (priority/ordinary/subordinated)

**Content Reorganization**:
1. **Logical restructuring**: Organize per client reading preference (often conclusion → process)
2. **Highlight key findings**: Emphasize critical discoveries and risk alerts
3. **Professional terminology**: Maintain legal terms but add brief explanations for complex concepts
4. **Readability**: Use client-friendly phrasing while preserving technical accuracy

### Step 3.5: Format Conversion Preparation (CRITICAL STEP)

**⚠️ MANDATORY: This step prevents format compliance failures**

**Context**: Technical reports (事实核查报告, 债权分析报告) use Markdown format for readability. Client deliverables MUST use pure text paragraph format. This conversion is REQUIRED, not optional.

**Format Analysis**:

Typical Markdown patterns in technical reports:
```markdown
## 二、合同签订情况           ← Markdown heading (禁止)

**主合同**：...                ← Bold syntax (禁止)

- 合同编号：XYZ-001          ← Bullet list (禁止)
- 金额：1,000,000.00元       ← Bullet list (禁止)
```

Required conversion to pure text:
```
二、合同签订情况              ← Plain text heading (允许)

2024年1月1日，债务人...      ← Complete sentence (允许)
```

**Conversion Guidelines**:

| Markdown Element | Conversion Rule | Example |
|-----------------|----------------|---------|
| ## Heading | Remove ##, keep text | `## 二、合同签订` → `二、合同签订情况` |
| ### Subheading | Integrate into paragraph | `### 主合同` → `关于主合同，...` |
| **Bold** | Remove asterisks | `**金额**：100万` → `金额为100万元` |
| - Bullet item | Convert to sentence | `- 编号：001` → `合同编号为001` |
| Multiple bullets | Combine into paragraph | See detailed example below |

**Detailed Conversion Example**:

Before (Markdown format):
```markdown
## 三、合同履行情况

**放款情况**：

- 2024年1月5日，债务人向债权人转账500,000.00元
- 2024年2月10日，债务人向债权人转账500,000.00元
- 累计放款金额1,000,000.00元

**还款情况**：

- 2024年6月30日，债权人收到还款100,000.00元
- 剩余欠款900,000.00元
```

After (Pure text format):
```
三、合同履行情况

2024年1月5日，债务人向债权人转账500,000.00元。2024年2月10日，债务人向债权人转账500,000.00元。累计放款金额1,000,000.00元。2024年6月30日，债权人收到还款100,000.00元，剩余欠款900,000.00元。
```

**Conversion Workflow**:

```
Step 1: Extract content from technical reports (with Markdown)
Step 2: Identify Markdown patterns (##, ###, -, **, etc.)
Step 3: Convert EACH pattern:
  - Headings: Remove symbols, keep hierarchy structure
  - Bullets: Convert to complete sentences with proper punctuation
  - Bold: Remove formatting, keep emphasis through context
  - Tables: Convert to narrative description (if any)
Step 4: Verify NO Markdown symbols remain
Step 5: Apply client format standards (dates, amounts, names)
```

**Pre-Generation Checklist**:
```
□ Identified all Markdown patterns in source content
□ Prepared conversion strategy for each pattern
□ Ready to generate pure text output
□ Verification commands prepared (grep checks)
```

#### ⚠️ CRITICAL: Template Format Is The Final Format

**Key Principle**: The template file itself IS in pure text format. DO NOT convert it.

**Template Format (Lines 67-121 in review_opinion_form_template.md)**:
```
[债权人名称]
债权审查意见

一、债权申报情况
...
```

**This IS the correct final format**:
- Title line 1: `[债权人名称]` (plain text, NO `#`)
- Title line 2: `债权审查意见` (plain text, NO `#`)
- Chapter titles: `一、债权申报情况` (plain text, NO `##`)

**⚠️ COMMON ERROR - DO NOT DO THIS**:
```markdown
# [债权人名称]债权审查报告   ← WRONG! Agent added `#`
```

**✅ CORRECT - Copy template format exactly**:
```
[债权人名称]债权审查报告    ← Correct! Plain text as in template
```

**Instruction**: When generating final report, use the EXACT SAME format as the template. Do NOT add any Markdown syntax (`#`, `##`, `-`, `**`). The template format IS the final format.

- ❌ Do NOT modify original calculation results

- **Structure**: Follow template structure (章节安排)

**Maintain audit trail**:
- Keep original technical reports in `工作底稿/`
- File inventory clearly lists all files
- Content can be traced back to source reports
- Calculation files preserved

### Rule 4: Client Delivery Standards

**Final deliverables must**:
- Conform to client template format
- Use client-specified terminology
- Follow client file naming conventions
- Include all required attachments

## Error Prevention & Prohibited Improvements

### ⚠️ CRITICAL: The "Well-Intentioned Improvement" Problem

**核心问题**: Report organizers often feel the urge to "improve" technical reports by:
- Clarifying ambiguous wording
- Filling in missing details
- Harmonizing inconsistencies
- Correcting perceived errors
- Making reports "read better"

**Why This Is Prohibited**:
- Report organizer is NOT an analyst - cannot verify accuracy of changes
- "Improvements" may introduce errors or change meanings
- Technical reports reflect actual evidence limitations (ambiguity may be intentional)
- Client needs unfiltered view of what analysts found

**Golden Rule**: **"纯搬运、不改善"** (Pure Transfer, No Improvement)

---

### 禁止改善清单 (Prohibited Improvements List)

#### ❌ 禁止改善 1: 澄清模糊表述 (Clarifying Ambiguity)

**What it is**: Technical report has intentionally vague language; organizer "clarifies" it

**Example**:
```
Technical Report: "利息约定不明,合同仅约定'按银行利率',未明确LPR期限"

❌ Wrong: "利息按1年期LPR计算"
   → Removed ambiguity, made definite choice

✅ Correct: "利息约定不明,合同仅约定'按银行利率',未明确LPR期限"
   → Preserved original ambiguity
```

**Why prohibited**: Ambiguity may reflect genuine evidence limitation; organizer cannot independently determine "correct" interpretation

---

#### ❌ 禁止改善 2: 补全不完整信息 (Completing Incomplete Information)

**What it is**: Technical report has gap; organizer fills it using "logic" or "context"

**Example**:
```
Technical Report: "放款日期: [证据未记载]"

❌ Wrong: "放款日期为合同签订后3天,即2020年1月4日"
   → Used contract date + "logic" to fill gap

✅ Correct: "放款日期: [证据未记载]"
   → Preserved gap notation
```

**Why prohibited**: "Logical" completion may be wrong; only analysts can fill gaps after verifying evidence

---

#### ❌ 禁止改善 3: 协调报告间不一致 (Harmonizing Inconsistencies)

**What it is**: Two technical reports have contradictory info; organizer "resolves" conflict

**Example**:
```
Fact Report: "放款日期 2020-01-05"
Analysis Report: "放款日期 2020-01-10"

❌ Wrong: "放款日期为 2020-01-10" (选择了看起来更准确的)
   → Organizer made choice between conflicting info

✅ Correct: "[注意: 事实报告记载1月5日,分析报告记载1月10日,存在不一致,需核实]"
   → Flagged inconsistency for resolution
```

**Why prohibited**: Organizer cannot determine which is correct; must flag for analyst review

---

#### ❌ 禁止改善 4: 优化语句流畅性 (Improving Sentence Flow)

**What it is**: Technical report has awkward phrasing; organizer rewrites for "better flow"

**Example**:
```
Technical Report: "利息计算结果为50万,但债权人申报60万,就低原则,确认50万"

❌ Wrong: "利息确认50万元"
   → Simplified for "smoothness", lost reasoning chain

✅ Correct: "利息计算结果为50万,但债权人申报60万,依据就低原则,确认50万"
   → Preserved complete reasoning (may add "依据" for clarity, but kept full logic)
```

**Why prohibited**: Simplifying may hide important qualifications or reasoning steps

---

#### ❌ 禁止改善 5: 添加上下文解释 (Adding Context)

**What it is**: Technical report makes bare statement; organizer adds "helpful" context

**Example**:
```
Technical Report: "诉讼时效已过"

❌ Wrong: "诉讼时效已过,债务人可以提出时效抗辩,债权人将面临败诉风险"
   → Added legal consequences not in original

✅ Correct: "诉讼时效已过"
   → Used exact wording from technical report
```

**Why prohibited**: Added context may be wrong or misrepresent analyst's intention; let analyst's words speak

---

#### ❌ 禁止改善 6: 统一术语 (Standardizing Terminology)

**What it is**: Technical reports use different terms for same thing; organizer "standardizes"

**Example**:
```
Fact Report uses: "欠条"
Analysis Report uses: "借款凭证"

❌ Wrong: Uniformly change both to "借款凭证" for consistency
   → Altered original terminology

✅ Correct: Keep "欠条" where fact report said it, "借款凭证" where analysis report said it
   → Preserved each report's terminology
```

**Why prohibited**: Different terms may reflect different evidence types or legal meanings; don't homogenize

---

#### ❌ 禁止改善 7: 纠正"笔误" (Correcting "Typos")

**What it is**: Organizer spots apparent typo/error and "fixes" it

**Example**:
```
Technical Report: "2020年1月1日" appears in one place, "2020-01-01" in another

❌ Wrong: Standardize all to "2020年1月1日" format
   → Made formatting changes

✅ Correct: Keep whatever format each source used (unless it's actually the template requirement to standardize)
   → Only standardize if template explicitly requires it

Technical Report: "借款100万元" (suspected typo, should be 1000万?)

❌ Wrong: "Correct" to "借款1000万元"
   → Changed substantive content

✅ Correct: Use exactly "借款100万元", flag: "[注意: 原文100万,如有误请分析员核实]"
   → Preserved original, flagged for review
```

**Why prohibited**: What looks like typo may be correct; only analysts can verify against source evidence

---

#### ❌ 禁止改善 8: 删除重复信息 (Removing Repetition)

**What it is**: Technical reports repeat same info; organizer removes "redundancy"

**Example**:
```
Both reports mention: "担保人为XX公司法定代表人"

❌ Wrong: Remove from one section to avoid repetition
   → Made editorial decision to delete

✅ Correct: Include in both places if both source reports mentioned it
   → Preserved structure of source reports
```

**Why prohibited**: Repetition may be intentional emphasis; organizer shouldn't make editorial judgments

---

#### ❌ 禁止改善 9: 调整内容顺序 (Reordering Content)

**What it is**: Organizer rearranges content for "better logic flow" beyond template requirements

**Example**:
```
Technical Report presents: A → B → C → D
Organizer thinks: D → A → B → C makes more sense

❌ Wrong: Rearrange to D → A → B → C
   → Changed presentation order

✅ Correct: Keep A → B → C → D (unless template explicitly requires different order)
   → Follow source report structure, adjust only as template requires
```

**Why prohibited**: Original order may reflect evidence chronology or analytical logic; don't second-guess

---

#### ❌ 禁止改善 10: 删除保留性表述 (Removing Hedging Language)

**What it is**: Technical report uses cautious phrasing; organizer makes it "more definitive"

**Example**:
```
Technical Report: "该证据疑似为放款凭证,但摘要栏为'往来款',性质待确认"

❌ Wrong: "该证据为放款凭证"
   → Removed "疑似" and qualification

✅ Correct: "该证据疑似为放款凭证,但摘要栏为'往来款',性质待确认"
   → Preserved all hedging language
```

**Why prohibited**: Hedging reflects genuine uncertainty; removing it misrepresents confidence level

---

#### ❌ 禁止改善 11: 添加过渡句 (Adding Transitions)

**What it is**: Organizer adds connecting sentences to make report "flow better"

**Example**:
```
Technical Report:
  Section A: "合同签订日期 2020-01-01"
  Section B: "违约发生日期 2020-12-31"

❌ Wrong: Add transition: "合同签订后,双方正常履约11个月,至2020年12月31日发生违约"
   → Added "正常履约11个月" narrative not in source

✅ Correct: Keep sections as is, or at most: "合同签订日期 2020-01-01。违约发生日期 2020-12-31。"
   → Direct statement without added narrative
```

**Why prohibited**: Transitions may imply facts not in evidence; just state facts sequentially

---

#### ❌ 禁止改善 12: 格式美化 (Format Beautification)

**What it is**: Organizer changes number formats, adds thousand separators, etc. beyond template requirements

**Example**:
```
Technical Report: "100万元"

❌ Wrong: Change to "1,000,000.00元" for "professionalism"
   → Changed format arbitrarily

✅ Correct:
  - If template requires "1,000,000.00元" format → Change it (template requirement)
  - If template allows either → Keep "100万元" (preserve original)
```

**Why prohibited**: Unnecessary format changes may introduce transcription errors; only change if template requires

---

#### ❌ 禁止改善 13: 替换专业术语为通俗语言 (Simplifying Technical Terms)

**What it is**: Organizer replaces legal terms with "client-friendly" language

**Example**:
```
Technical Report: "该担保构成越权担保"

❌ Wrong: "该担保未经过必要的公司决议程序"
   → Replaced legal term with plain description

✅ Correct: "该担保构成越权担保"
   → Kept legal terminology
```

**Why prohibited**: Legal terms have precise meanings; "simplifying" may change meaning or lose precision

---

#### ❌ 禁止改善 14: 移除缺失标注 (Removing Gap Markers)

**What it is**: Technical report marks info as missing; organizer removes marker to "clean up"

**Example**:
```
Technical Report: "还款日期: [证据未记载]"

❌ Wrong: Delete this line (seems incomplete/unprofessional)
   → Removed gap marker

✅ Correct: "还款日期: [证据未记载]"
   → Preserved gap marker
```

**Why prohibited**: Gap markers inform client of evidence limitations; hiding them misrepresents completeness

---

#### ❌ 禁止改善 15: 调整风险提示级别 (Adjusting Risk Alert Levels)

**What it is**: Technical report flags high-risk item; organizer softens it to avoid "alarming client"

**Example**:
```
Technical Report: "⚠️ 重大风险: 该担保可能无效"

❌ Wrong: "该担保效力存在争议"
   → Downgraded risk level

✅ Correct: "⚠️ 重大风险: 该担保可能无效"
   → Preserved risk alert exactly
```

**Why prohibited**: Risk assessment is analyst's professional judgment; organizer cannot modify severity

### 自查问题 (Self-Check Questions)

**Before finalizing report, ask yourself:**

**Q1: 是否有任何一个句子在两份技术报告中都找不到对应原文?**
- ✅ No → Safe (all content sourced)
- ❌ Yes → Review: Is this sentence added by me? If yes, remove it

**Q2: 是否有任何地方我"选择"了两个不一致信息中的一个?**
- ✅ No → Safe (no arbitration)
- ❌ Yes → Review: Have I flagged the inconsistency instead of choosing?

**Q3: 是否有任何地方我"填补"了技术报告的空白?**
- ✅ No → Safe (no gap-filling)
- ❌ Yes → Review: Have I preserved gap markers instead?

**Q4: 是否有任何地方我"改善"了技术报告的表述?**
- ✅ No → Safe (pure transfer)
- ❌ Yes → Review: Is this improvement required by template? If not, revert to original

**Q5: 是否有任何地方我删除了技术报告的保留性/模糊性表述?**
- ✅ No → Safe (preserved uncertainty)
- ❌ Yes → Review: Restore hedging language

**Q6: 能否随机抽取5个段落,在技术报告中找到对应原文?**
- ✅ Yes → Safe (high traceability)
- ❌ No → Review: Why can't I find source? Did I add/modify content?

### Mandatory Actions (Zero-Tolerance Items)

**These are NEVER acceptable - Zero exceptions:**

1. ❌ **Modify calculations or amounts** from source reports
2. ❌ **Add new conclusions** not in technical reports
3. ❌ **Remove risk alerts or qualifications**
4. ❌ **Change debt classifications** (priority/ordinary/subordinated)
5. ❌ **Alter dates** (except formatting per template requirement)
6. ❌ **Resolve inconsistencies** between technical reports independently
7. ❌ **Delete gap markers** ([证据未记载], [待补充], etc.)
8. ❌ **Modify legal characterizations** (越权担保, 诉讼时效, etc.)
9. ❌ **Change calculation results** or interest amounts
10. ❌ **Add explanations** not in technical reports

**Verification**: Before delivery, verify amounts, dates, conclusions, and all gap markers match source reports exactly.

**Violation Protocol**: If ANY prohibited improvement is detected during review, report is rejected and must be regenerated from scratch.
